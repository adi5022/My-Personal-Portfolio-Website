# Site Structure and Evaluation Rules

## Purpose

This document defines the baseline structure of the website and the mandatory evaluation process that must occur before implementing new ideas, features, animations, or design changes.

The objective is to ensure that creativity, performance, maintainability, and user experience remain balanced throughout development.

---

# Rule 1: Single Page Narrative Flow

The website should function as a guided experience.

Users should naturally progress through sections by scrolling.

The experience should feel connected rather than a collection of unrelated pages.

Additional pages may exist when necessary, but the primary experience should remain scroll-driven.

---

# Rule 2: Hero Section

The first screen presented to visitors must contain:

* Name
* Primary identity
* Core introduction

This section acts as the opening statement of the website.

The hero section should:

* Immediately communicate who the individual is.
* Establish the visual theme.
* Set expectations for the remainder of the experience.

The hero section should not be overcrowded.

---

# Rule 3: Personal Introduction Section

Following the hero section, visitors should learn more about the individual.

This section may include:

* Background
* Story
* Journey
* Goals
* Personal philosophy

The focus should be on creating a human connection.

---

# Rule 4: Skills Section

A dedicated skills section should exist.

Skills may be categorized into groups such as:

* Technical skills
* Creative skills
* Professional skills
* Academic skills

The presentation should prioritize clarity over visual complexity.

---

# Rule 5: Interests Section

A dedicated section should describe interests.

Examples include:

* Technology
* Research
* Art
* Design
* Gaming
* Music
* Writing
* Learning

The goal is to present personality beyond professional capabilities.

---

# Rule 6: Hobbies Section

A dedicated section should exist for hobbies and activities.

The purpose is to increase relatability and authenticity.

This section should remain lightweight and easy to browse.

---

# Rule 7: Project and Work Sections

Projects should be presented as evidence of capability.

Each project should communicate:

* Problem
* Approach
* Outcome
* Technologies
* Lessons learned

Projects should not be listed without context.

---

# Rule 8: Contact Section

The website should conclude with a clear contact area.

Visitors should always know how to reach out or continue exploring.

---

# Rule 9: Theme Consistency

Every section must support the active project theme.

The theme should influence:

* Visual language
* Typography
* Motion
* Content presentation
* Layout decisions

No section should feel disconnected from the overall experience.

---

# Rule 10: Controlled Complexity

Complexity should only be added when it creates measurable value.

Complexity should never exist solely because it is technically impressive.

The website should prioritize:

1. Clarity
2. User experience
3. Performance
4. Visual quality

in that order.

---

# Rule 11: Feature Evaluation Requirement

Before implementing any new feature, animation, interaction, effect, or visual concept, the AI agent must perform an evaluation.

Implementation may only occur after the evaluation is reported.

---

# Rule 12: Mandatory Evaluation Report

Every proposed feature must receive a report containing:

## Description

What the feature does.

## Benefits

Why the feature may improve the website.

## Drawbacks

Potential disadvantages.

## Technical Complexity

Rating:

* Very Low
* Low
* Medium
* High
* Very High

## Maintenance Cost

Rating:

* Very Low
* Low
* Medium
* High
* Very High

## Performance Impact

Rating:

* Negligible
* Minor
* Moderate
* Significant
* Severe

## Accessibility Impact

How the feature affects accessibility.

## Mobile Impact

How the feature affects mobile users.

## Deployment Impact

Whether the feature introduces deployment complications.

---

# Rule 13: Alternative Solution Requirement

The AI agent must never provide only one solution.

Every proposal should include:

### Option A

Requested approach.

### Option B

Simpler alternative.

### Option C

Performance-optimized alternative.

The tradeoffs between options must be explained.

---

# Rule 14: Performance Budget Protection

The AI agent must actively protect website performance.

Features likely to impact:

* Loading speed
* Rendering speed
* Responsiveness
* Animation smoothness

must be clearly identified before implementation.

---

# Rule 15: Animation Review Requirement

Every animation must be evaluated according to:

* Purpose
* Performance cost
* Accessibility impact
* Mobile compatibility
* Browser compatibility

Animations that fail evaluation should be rejected.

---

# Rule 16: SVG Motion Preference

When motion is required:

1. SVG animations should be preferred.
2. CSS animations should be considered next.
3. JavaScript animation libraries should be used only when justified.

The simplest effective solution should be preferred.

---

# Rule 17: User Experience Protection

The AI agent must reject any feature that significantly harms:

* Readability
* Navigation
* Discoverability
* Responsiveness

even if the feature appears visually impressive.

---

# Rule 18: Continuous Site Auditing

Throughout development, the AI reviewer must continuously assess:

* Layout quality
* Responsive behavior
* Accessibility concerns
* Performance concerns
* Animation quality
* Technical debt
* Consistency with theme

Findings should be reported before major implementation phases.

---

# Rule 19: Recommendation System

When the user proposes an idea, the AI should not simply implement it.

The AI must:

1. Analyze the idea.
2. Assess complexity.
3. Assess performance impact.
4. Assess maintainability.
5. Suggest alternatives.
6. Recommend the most appropriate solution.

Only then should implementation begin.

---

# Rule 20: Long-Term Maintainability

Every decision should be evaluated according to the question:

"Will this still be easy to maintain, modify, and extend six months from now?"

If the answer is no, an alternative approach should be proposed.
